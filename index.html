How to Get Started

Contributing to open source can be intimidating:
  • Any code that you submit is open to the review and criticism of the project maintainers and the public.
  • You might be afraid that people in the project won't be nice
    • Linus Torvalds is known for commenting on one pull request to the Linux kernel, "Christ people. This is just sh*t."
  • There are barriers to entry in learning what and how to do it

Tools to Get Started

  • npm
  • git & github
  • testing framework
  • node
The tools for open source are primarily the tools used by most modern developers. However, each community will be different. Many projects will require tests to be submitted with any pull request.

Finding a Project

Choose a Project You Use on a Regular Basis

  • if you don't, you may be capable of making documentation updates and simple bug fixes
  • you won't have the context needed for making larger bug fixes, refactors and new features
    • open source projects are a community
    • large changes are commonly discussed before they're implemented
    • you won't know the roadmap for larger changes and features

Small Projects

  • Small projects are often maintained by a few people at most.
    • maintainers may be happy to accept help
    • but it may also mean that they're too busy to provide help
  • it's easier to wrap your head around the whole project.
    • this may mean that the maintainers already have a specific vision for the project.

Large Projects

  • Large projects may be maintained by a large team of people
    • they have time to flag beginner issues and provide feedback
    • may also mean that they're overwhelmed by pull requests
  • harder to understand the entirety of a large project
    • may be easier to find a small piece that you can work
  • large amounts of documentation to work on

Making Your First Commit

fork the repo

You may want to make a new branch, if you want to keep your changes out of the master version of your fork

make your changes

create or fix tests

create any necessary tests for new code and make sure existing tests pass

commit your code

  • if you only make one commit to your branch, the commit message should give a brief summary of the changes that you made
  • if you make multiple commits to your branch, the project may ask you to use `git rebase` to collapse those commits into one commit that explains your contribution

create a pull request

  • usually your PR will be against the master branch of the parent repo
  • the project will probably ask that your PR contain a summary of your changes, tests, screenshots of any UI changes, etc

sign the Contributor License Agreement (CLA)

A legal document, usually required by larger companies, that signs away rights to the code you contributed

What to Expect Next

  • Once you submit your pull request, it is up to the maintainers.
  • automated tests
    • some of these may be private tests that you haven't seen
  • one or more code reviews
  • your PR may be submitted to the larger community for comments
  • if you don't hear anything, it is appropriate to ping the project maintainers after a reasonable amount of time

Growing as a Contributor

One common path to growth seems to be

documentation fixes or improvements

minor bug fixes

larger bug fixes or features

ability to review pull requests and merge

project maintainer

job developing open source
speaking or consulting
burnout
Like any volunteer position, burnout is a possibility. I think one of the keys to avoiding burnout is keeping a sane schedule for yourself.
